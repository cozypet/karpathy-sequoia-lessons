Human-Legible vs. Agent-Legible
Agent-native is not a documentation format. It is a system surface redesign.
Human-Legible
"Go to Settings > Pages"
Screenshot-based tutorials
Prose error messages
Multi-step GUI workflows
Implicit state in browser sessions
Agent-Legible
Structured API with typed errors
Machine-parseable config schemas
Error codes with recovery hints
Single-call idempotent operations
Explicit state in request/response
The design question: can an agent reason about this without hallucinating missing context?

Encoding Taste Into Constraints
Taste doesn't scale through hiring. It scales through constraint surfaces that agents can't violate.
Fragile
Relies on humans
Verbal review
"Don't use email as user ID" spoken in standup
Senior oversight
One person catches all the Stripe/Google bugs
Durable
Written specs
Design docs
"All identity must use persistent UUID, never email"
Architecture rules
Written constraints in team wiki
Enforced
Agent-facing
System constraints
Schema rejects email as foreign key at validation layer
Behavioral contracts
Tests fail if agent generates email-based identity lookup
Karpathy's Stripe/Google email bug would never ship if "use persistent user IDs" was a constraint, not a preference.

08 / 08
The Architect's Checklist
Five questions to ask before your next agentic system ships.
Have I mapped which tasks fall inside vs. outside my model's strong RL circuits?
Is my verification loop automated, or does it still require a human reading output?
Can my agents reason about my system surfaces without hallucinating missing context?
Are my team's taste decisions encoded as constraints, or trapped in someone's head?
Is my team investing in domain understanding, or just chasing better model access?
If you answered "no" to any of these, you have architecture debt. The model won't fix it for you.
